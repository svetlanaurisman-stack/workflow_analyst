Универсальный workflow системного аналитика

Типичный путь работы с требованиями. Выберите вашу методологию ниже, чтобы адаптировать результаты:

1. Сбор требований и определение объема проекта
🎯 Цель: Сформировать первоначальный перечень требований и гипотез, определить границы проекта, выявить ключевых стейкхолдеров.

✔️ Действия:

  • Провести воркшопы с заказчиком/продукт-менеджером и стейкхолдерами.
  • Собрать требования (User Stories, Jobs To Be Done, Use Cases и т.д.).
  • Использовать техники (Event Storming, интервью, опросы).
  • Совместно с архитектором разработать концептуальную архитектуру/HLD.
⛳ Результат:
Верхнеуровневый бэклог продукта.
Основа для Устава проекта (Project Charter) / Спецификация требований высокого уровня.
📌 Независимо от подхода необходимо определить ключевых стейкхолдеров и стратегию.
2. Приоритизация требований и планирование релизов/этапов
🎯 Цель: Определить порядок реализации требований, выделить MVP, построить план релизов или этапов проекта.

✔️ Действия:

  • Применить техники приоритизации (MoSCoW, Value vs Effort, RICE/ICE и др.).
  • Согласовать состав MVP.
  • Построить дорожную карту релизов на 3-6 месяцев (Agile) / План этапов проекта (Waterfall).
  • Оценить объем аналитических работ по приоритетным задачам.
⛳ Результат:
Приоритизированный бэклог продукта с выделенным MVP и дорожной картой.
Утвержденный Устав проекта с этапами и высокоуровневым объемом для каждого / Приоритизированный план детализации требований по этапам.
3. Анализ, уточнение и документирование требований
🎯 Цель: Подготовить требования для реализации.

✔️ Действия:

  • Провести интервью, воркшопы, анкетирование.
  • Использовать методы: опросники, мозговые штурмы, анализ существующей документации.
  • Зафиксировать функциональные и нефункциональные требования.
  • 🏃 Часто используется декомпозиция User Stories на задачи (Tasks), определяются Acceptance Criteria.
  • Создать прототипы интерфейсов (Figma, Balsamiq) для визуализации.
  • Построить диаграммы для визуализации процессов.
  • Устранить конфликты требований через согласование с заказчиком.
  • Верифицировать и валидировать требования.
⛳ Результат:
Готовые к работе задачи в бэклоге спринта.
Готовые к работе утвержденные спецификации требований.

📌 Потребуется явно организовать подписание документации, чтобы формально зафиксировать минимальность дальнейших изменений. Требования могут быть представлены в виде User Story, но объединены в общую спецификацию.
4. Управление изменениями требований
🎯 Цель: Минимизировать риски несвоевременных изменений.

✔️ Действия:

  • Настроить процесс запроса изменений (Change Request).
  • Оценивать влияние изменений на сроки, бюджет и архитектуру.
  • Обновлять документацию и уведомлять команду.
⛳ Результат:
Изменения неизбежны. Три провальных подхода: игнорирование, принятие без анализа и попытки "законсервировать". Эффективная альтернатива — четко определенный процесс для оценки изменений ("правила игры") и доверие стейкхолдеров.
Изменения фиксируются в виде спецификаций и встроены в общий бэклог. Даже после согласования базовой версии (baseline) могут возникнуть новые факторы. Каждое изменение рассматривается и приоритизируется.
5. Передача требований в разработку
🎯 Цель: Обеспечить четкое понимание требований командой.

✔️ Действия:

  • Провести встречу с разработчиками, тестировщиками и архитекторами.
  • 🏃 В Agile - это Backlog Refinement, Sprint Planning.
  • Ответить на вопросы и уточнить спорные моменты.
  • Обеспечить доступ к документации.
⛳ Результат:
Команда приступает к реализации спринта.
Команда приступает к реализации этапа.
6. Контроль качества и сопровождение
🎯 Цель: Следить за соответствием продукта исходным требованиям.

✔️ Действия:

  • Ежедневные встречи для синхронизации команды.
  • Участвовать в тестировании для проверки критериев приемки (SRS).
  • Вносить корректировки требований по мере разработки.
  • Фиксировать отклонения и инициировать корректирующие действия.
  • Подготовить демо/Sprint Review для стейкхолдеров.
  • 🌊 В Waterfall аналитик участвует в отдельной фазе Приемочного тестирования.
⛳ Результат: Продукт соответствует утвержденным требованиям.
7. Внедрение продукта
🎯 Цель: Завершить работу с требованиями и передать продукт заказчику.

✔️ Действия:

  • Участвовать в разработке релизной стратегии и пилотирования.
  • Участвовать в подготовке отчетов и обучении пользователей.
  • Разработать структуру базы знаний, написать инструкции.
  • Участвовать в поддержке (3 линия) для разбора сложных инцидентов.
⛳ Результат: Успешная сдача рабочего продукта, готового к эксплуатации.
